## Key Learnings

### GitHub 101 Lesson Module

**Core Concepts I Now Understand:**
- **Repositories (Repos):** Project folders with built-in version history that track all file changes
- **Branches:** Parallel copies allowing independent work without disrupting the main codebase
- **Commits:** Snapshots of work with descriptive messages - should commit frequently when something works
- **Pull Requests (PRs):** Formal review process before merging changes, creating accountability and preventing errors
- **Push/Pull:** Upload local work to GitHub vs. download latest changes from GitHub

**Practical Workflow (7-Step Process):**
1. Switch to or create branch with latest changes
2. Edit files as needed
3. Commit meaningful chunks of work
4. Push commits to GitHub
5. Open pull request for review
6. Address feedback and update branch
7. Merge approved changes and delete branch

**Advanced Techniques:**
- **Stashing:** Temporarily save uncommitted changes to switch branches safely
- **Rebasing vs. Merging:** Rebase for solo branches (clean history), merge for shared branches (safer)
- **File Organization:** Keep repos outside cloud storage to prevent sync conflicts

## Notes

- GitHub isn't just for code - useful for any collaborative knowledge work (docs, playbooks, specs)
- PRs create accountability and catch errors before they reach production
- Commit frequently as safety checkpoints, especially before risky changes
- Clear PR descriptions and appropriate reviewer selection are crucial for effective collaboration
